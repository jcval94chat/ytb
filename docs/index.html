Análisis técnico integral del repositorio

Pipeline batch en Python que extrae metadatos de videos de YouTube por bloques semanales y sobrescribe un dataset consolidado en Google Sheets.

Tipo: ETL/Integración batch Lenguajes: Python + YAML Runtime: Python 3.12.7 + GitHub Actions Complejidad: Baja-Media Confianza del análisis: Alta (código + workflows revisados)

1) Resumen ejecutivo

El repo implementa un proceso automático semanal que toma un subconjunto rotativo de canales de YouTube, obtiene sus videos recientes (últimos 60 días), agrega métricas (views/likes/comments), combina con el contenido existente en una hoja de cálculo y escribe todo de nuevo en Google Sheets. El flujo principal está en youtube_data.py y la lógica de partición semanal en get_urls.py. El disparo operativo real está en .github/workflows/youtube_data.yml.

Core: extracción YouTube + normalización + merge + escritura en Sheets. Auxiliar: keepalive de repositorio, README incompleto, y archivo heartbeat.

3) Flujo completo del dato (end-to-end)

  1. Trigger: cron lunes 00:00 UTC o ejecución manual del workflow youtube_data.yml.
  2. Configuración: carga secretos YOUTUBE_API_KEY, GOOGLE_SHEETS_CREDS_BASE64, SPREADSHEET_ID y opcionales REFERENCE_DATE/FORCE_CHUNK_INDEX.
  3. Selección de canales: get_urls() elige un bloque de ~40 canales según semana ISO o bloque forzado.
  4. Resolución de canal: para cada URL, se obtiene channel_id y nombre (casos /channel/, /user/, @handle + fallback por búsqueda).
  5. Extracción: search().list pide videos desde publishedAfter (60 días), pagina con nextPageToken, luego videos().list trae snippet/contentDetails/statistics.
  6. Transformación: duración ISO8601→segundos, fecha normalizada, casting de métricas a enteros, agrega execution_date.
  7. Persistencia: lee todos los registros de la hoja, concatena con los nuevos, dedup por video_id + channel_name, borra hoja y sube dataset completo.
  8. Errores/logs: logging a consola + youtube_data.log; el workflow sube ese log como artifact siempre.

  • python youtube_data.py: entrypoint principal local/CI.
  • Workflow youtube_data.yml: trigger real de producción (schedule + workflow_dispatch con inputs).

6) Interfaces y movimiento de datos

Interfaces detectadas

  • YouTube Data API v3: endpoints de búsqueda/canales/videos vía googleapiclient.discovery.build.
  • Google Sheets API (vía gspread): lectura completa, limpieza y actualización total de sheet1.

Contratos de datos

  • Registro de video: channel_name, video_id, title, description, upload_date, tags, thumbnail_url, duration_seconds, views, likes, comments, execution_date.
  • Deduplicación por clave compuesta video_id + channel_name.
  • No hay esquema versionado formal (JSON Schema/Avro/etc.). Inferencia: riesgo de drift de columnas.

7) Ejecución y despliegue

  • Tests: no hay suite de tests automatizados en el repo.
  • Despliegue/operación: GitHub Actions sobre ubuntu-latest, Python 3.12.7, cron semanal + manual.
  • Infra adicional: no se detectan Dockerfile, compose, Terraform, Kubernetes ni scripts de release.
  • Variables críticas: YOUTUBE_API_KEY, GOOGLE_SHEETS_CREDS_BASE64, SPREADSHEET_ID, opcionales REFERENCE_DATE, FORCE_CHUNK_INDEX.

8) Dependencias importantes

  • google-api-python-client: cliente para YouTube API.
  • gspread + google-auth: autenticación service account y operaciones de hoja.
  • pandas: consolidación tabular y deduplicación.
  • isodate: parseo de fechas/duraciones ISO.

9) Caso práctico narrado

Un lunes se ejecuta el workflow. Si no hay override, la semana ISO define el bloque de canales. Para cada canal, el script resuelve su ID, obtiene videos publicados en 60 días, normaliza métricas y duración, acumula filas, une con histórico de Google Sheets y elimina duplicados. Finalmente, borra la hoja y la vuelve a poblar completa. El equipo consumidor puede seguir leyendo una sola hoja consolidada con información reciente y acumulada.

10) Riesgos y puntos ciegos

Hecho confirmado: el script ejecuta sheet.clear() antes de escribir; una falla posterior puede dejar la hoja vacía o parcial.
Hecho confirmado: no hay tests automáticos; cambios en parsing/API pueden romper en runtime.
Hecho confirmado: manejo de @handle usa forUsername y fallback search; puede devolver canal incorrecto si la búsqueda es ambigua.
Inferencia: posible presión de cuota API por alto volumen de canales y paginación por canal.
Inferencia: duplicidad de fuente de canales (función get_channels() y CHANNEL_URLS) puede introducir drift funcional.
